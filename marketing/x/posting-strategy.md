## Purpose

This file turns `marketing/x/playbook.md` into practical posting strategy for the founder's personal X account.

Use this file to decide:

- which strategic content motion a post serves
- which post category to use
- which lightweight format fits the idea
- what the near-term posting cadence should be
- what the first week of consistent posting should focus on

This file owns post motions, post types, post formats, cadence, and first-week execution.

This file does not own:

- Orisen's product truth
- public claim boundaries
- cross-channel founder voice

- detailed reply workflow
- finished post drafts
- finished hooks

## Strategic content motions

Each post should serve one of these motions.

Do not post random startup, wellness, productivity, or sleep content unless it connects back to one of these motions.

### 1. Build Orisen in public

Purpose:

- show that Orisen is real and actively being built
- build founder trust
- make the product feel tangible
- attract useful builders, operators, experts, and early users

Good topics:

- prototype progress
- product decisions
- hardware/software tradeoffs
- reliability constraints
- lessons from the old MVP
- what broke or changed
- what is harder than expected

### 2. Make the wake-up problem specific

Purpose:

- make high-pain users feel seen
- surface replies, DMs, and customer language
- avoid drifting into generic sleep wellness

Good topics:

- sleeping through alarms
- turning alarms off half-asleep
- waking up but not getting out of bed
- going back to bed after initially getting up
- losing time between alarm time and being out of bed
- not trusting yourself in the morning

### 3. Reframe the problem

Purpose:

- shift people from “I need more discipline” or “I need a louder alarm” to “waking up is a completion problem”
- make Orisen's wedge easier to understand
- build the tracking-to-intervention narrative carefully

Good topics:

- normal alarms stop too early
- alarm success is not wake completion
- trackers tell you what happened but do not change the morning
- the morning failure happens after the alarm rings
- systems should be designed around half-awake behavior

### 4. Learn from users and experts

Purpose:

- use X for customer discovery
- learn real language
- find high-pain users
- sharpen product assumptions before overbuilding the post bank

Good topics:

- what usually fails in the morning
- what users have tried already
- what feels too extreme
- what feels useful
- what people would tolerate from a wake-up device
- what experts think about validation or claim boundaries

## Practical post categories

Use this smaller set when choosing what to write.

These categories are broad enough for variety but narrow enough for GPT to choose from reliably.

| Category | Main job | Best used for |
|---|---|---|
| Pain mirror | Make target users feel seen | oversleeping, snoozing, waking but not getting up |
| Problem insight | Say the sharp observation clearly | alarm failure, wake completion, half-awake behavior |
| Reframe | Change how people see the problem | discipline vs system, alarm vs completion, tracking vs intervention |
| Build update | Show real progress | prototype, firmware, app, hardware, testing, design decisions |
| Product tradeoff | Build trust in founder judgment | reliability, privacy, sensing, no-camera decisions, offline behavior |
| Customer discovery ask | Learn from high-pain users | replies, DMs, interviews, pain language |
| Expert/operator ask | Learn from credible people | sleep, hardware, behavior change, validation, regulatory caution |
| Claim-safe vision | Show ambition without overclaiming | future sleep intervention, less brutal mornings, long-term product direction |
| Evidence/traction note | Share signal carefully | waitlist response, useful replies, learnings, not proof of PMF |

Do not try to use every category in one week.

For the early phase, prioritize:

1. pain mirror
2. problem insight
3. reframe
4. build update
5. customer discovery ask

## Week 1 focus

The immediate focus is to start posting consistently enough to learn, without pretending the strategy is final.

Week 1 should prioritize:

- learning what wake-up pain language gets replies, DMs, or waitlist interest
- making the wake-up failure pattern more specific
- showing that Orisen is being built by a real founder
- testing the wake-completion reframe in simple language
- using replies to find real conversations and customer language

Week 1 should not prioritize:

- a large posting system
- polished threads for every idea
- heavy investor-facing content
- detailed technical explanations before the pain is clear
- filling the full post bank before the first few posts are tested

The account should revolve around four core motions:

- build in public
- wake-up pain
- wake-completion reframe
- customer discovery

## Post formats

These are reusable structures, not finished drafts.

Finished post drafts belong in `marketing/x/post-bank.md`.

### Pain mirror

Use when the goal is to make the target user feel seen.

Structure:

1. Name the exact morning failure.
2. Make it emotionally specific.
3. Do not over-explain.

Example shape:

> Waking up is not always the hard part.
>
> Sometimes you wake up, turn the alarm off, and still lose the morning.

### Problem insight

Use when the goal is to make a sharp observation.

Structure:

1. Start with the insight.
2. Add one line of explanation.
3. Stop before it becomes a lesson.

Example shape:

> Normal alarms stop too early.
>
> They stop when you press a button, not when the wake-up is actually complete.

### Reframe

Use when the goal is to change how people think about the category.

Structure:

1. State the old frame.
2. Replace it with the new frame.
3. Keep the new frame simple.

Example shape:

> I do not think waking up is just an alarm problem.
>
> For some people, it is a completion problem.

### Build update

Use when the goal is to show real progress.

Structure:

1. Say what was worked on.
2. Say what made it hard or interesting.
3. Share one lesson or next step.

Example shape:

> Worked on the wake-completion flow today.
>
> The hard part is not making the alarm louder.
>
> It is deciding when the device should trust that you are actually out of bed.

### Product tradeoff

Use when the goal is to show product judgment.

Structure:

1. Name the tradeoff.
2. Explain why it matters for the user.
3. Keep the language practical.

Example shape:

> The alarm cannot depend on Wi-Fi to work.
>
> If the internet is down and the alarm fails, that is not a smart device. That is a bad alarm.

### Customer discovery ask

Use when the goal is to learn from high-pain users.

Structure:

1. Name a specific failure mode.
2. Ask one clear question.
3. Avoid sounding like a survey.

Example shape:

> If you set multiple alarms and still oversleep, what usually fails?
>
> Sleeping through it, turning it off, or getting back in bed?

### Expert/operator ask

Use when the goal is to attract feedback from credible people.

Structure:

1. Name the thing being figured out.
2. Say who would have useful feedback.
3. Ask simply.

Example shape:

> I am thinking through how to validate wake-completion behavior without overclaiming anything about sleep stages.
>
> Would be grateful to compare notes with anyone who has worked on sleep, behavior change, or consumer hardware.

### Claim-safe vision

Use when the goal is to show ambition without overstating evidence.

Structure:

1. Start from the validated/current wedge.
2. Say what Orisen is being built toward.
3. Use future framing clearly.

Example shape:

> The first wedge is wake completion.
>
> Longer term, I think sleep tech should move from only telling you what happened to helping change what happens next.

### Evidence/traction note

Use when the goal is to share early signal carefully.

Structure:

1. Say what happened.
2. Say what it suggests, not what it proves.
3. Tie it to the next learning step.

Example shape:

> The strongest replies so far are not about sleep tracking.
>
> They are about people not trusting themselves after the alarm goes off.
>
> That is the pain I want to understand better.

## Cadence and weekly operating rhythm

Keep the cadence lightweight enough for a solo founder to follow.

### Starting cadence

For the early phase, aim for:

- 1 original post per day, or 5 original posts per week if daily posting is not realistic
- 10 to 20 thoughtful replies per day on days when time allows
- 1 to 2 customer-discovery asks per week
- 1 to 2 build-in-public updates per week
- 1 lightweight weekly review

Do not force low-quality posts just to satisfy cadence.

A strong 5-post week is better than a weak 14-post week.

### Weekly mix

A simple starting mix:

- 2 pain/problem posts
- 1 reframe post
- 1 build update
- 1 customer-discovery or expert/operator ask

Add claim-safe vision posts only when they do not outrun the evidence.

Use direct waitlist CTAs sparingly.

Early X should create conversation and qualified intent, not just clicks.

### Weekly workflow

Once per week:

1. Pick 3 to 5 ideas from actual building, user conversations, or founder observations.
2. Turn each idea into one post category.
3. Draft posts using `marketing/x/voice.md`.
4. Check claims against `product/claims-and-evidence.md`.
5. Post and reply manually.
6. Save the best-performing or best-feeling drafts into `marketing/x/post-bank.md` later.
7. Log meaningful lessons in `marketing/post-performance-log.md` only when there is a real signal.

## First-week posting plan

The first week should be simple.

Goal:

> Learn which wake-up pain language creates real replies, DMs, profile clicks, or waitlist interest.

Do not try to look like a polished brand account.

### Day-by-day starter plan

| Day | Main post type | Goal |
|---|---|---|
| Day 1 | Pain mirror | Make the target user feel seen |
| Day 2 | Problem insight | Test the “alarm can work and morning can fail” idea |
| Day 3 | Build update | Show Orisen is being built by a real founder |
| Day 4 | Customer discovery ask | Learn exact wake-up failure modes |
| Day 5 | Reframe | Test wake completion as the sharper frame |
| Day 6 | Product tradeoff or build update | Build trust through product judgment |
| Day 7 | Light reflection or evidence note | Summarize what replies/DMs taught, without overclaiming |

### First-week reply focus

During week 1, reply under conversations about:

- sleeping through alarms
- setting many alarms
- turning alarms off and going back to sleep
- brutal mornings
- sleep trackers and alarm apps
- consumer hardware or behavior-change products

Detailed reply workflow belongs in `marketing/x/reply-strategy.md`.

For now, use the simple reply rhythm from `marketing/x/voice.md`:

> reflect the point + add one specific thought + ask one useful question if natural

### First-week CTA rhythm

Use CTAs lightly.

Good first-week CTAs:

- ask people what usually fails in the morning
- ask people what they have already tried
- invite high-pain users to DM if they repeatedly struggle with this
- share the waitlist only when the post clearly explains the product or someone asks for it

Avoid:

- putting the waitlist link in every post
- pitching in every reply
- asking broad, low-signal questions
- trying to sound bigger than the current stage

## What not to post yet

Avoid early posts that:

- claim Orisen eliminates grogginess, fixes sleep inertia, or guarantees perfect wake-ups
- lead with sleep-stage control, AI, radar, or technical claims before the pain is clear
- present early waitlist interest as product-market fit or paid demand
- make every post a waitlist pitch
- drift into generic productivity, generic wellness, or random founder content disconnected from Orisen's core motions
- over-index on investor-facing takes before customer pain is clear
- turn every idea into a long thread
- rely on generic engagement bait

## Maintenance rules

Keep this file practical.

Update this file when:

- weekly posting reveals which categories are easiest to sustain
- certain formats repeatedly produce target-customer replies
- cadence proves too heavy or too light
- `marketing/x/playbook.md` changes channel goals or account strategy
- `marketing/x/voice.md` changes X style guidance
- Orisen moves from waitlist/customer discovery into beta, preorder, or launch

Do not use this file to store:

- finished posts
- finished hooks
- finished replies
- detailed reply workflows
- detailed posting calendars
- raw performance logs
- raw customer conversations
- claim decisions

Move finished or near-finished assets to `marketing/x/post-bank.md` later.

Move detailed reply execution to `marketing/x/reply-strategy.md`.

Move scheduled posts to `marketing/x/posting-calendar.md` once there is a real schedule.
